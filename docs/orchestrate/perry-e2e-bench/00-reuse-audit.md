# 00 -- Reuse audit: E2E benchmarking harness (publish + artillery client)

Scope: building an E2E benchmarking harness that publishes a SpacetimeDB module (compiled both V8 and Perry AOT paths), then hammers it with a Rust artillery-style client measuring cold startup time and transactions per second across three load types: empty, router, cpu_heavy.

## Top reuse recommendation

The strongest reuse candidate is **`CompiledModule` + `perry_artifacts_dir` + the `benchmarks-ts` module sources** as a combined unit. The module compilation infrastructure (`CompiledModule::compile` for V8, `from_prebuilt` for Perry) and the existing `benchmarks-ts` reducers (`empty`, plus circles/ia_loop for cpu_heavy candidates) provide the "build both module variants" half of the E2E harness. The `docker-compose.yml` provides the server half. What does NOT exist and must be built greenfield is: (1) a Rust HTTP/WS artillery-style client that connects to the running server, calls reducers at concurrency, and measures TPS + latency percentiles; (2) a cold-startup measurement (timing from `spacetime publish` to first successful reducer call); (3) a "router" module shape with multiple reducers exercising dispatch-table overhead; (4) the multi-reducer Perry dispatch (extending `format_stdb_abi_shim_c` from single-reducer id==0 to an N-reducer switch). The existing in-process Criterion bench is not applicable -- it measures a fundamentally different thing (per-call latency without network, concurrency, or server process overhead).

## Borderline calls

- **SpacetimeDB bench crate (Criterion in-process):** classified "not applicable" because the E2E harness specifically requires network publish + external client + concurrency + cold startup. However, if the brief's "TPS" metric were redefined to mean "in-process reducer calls per second without network" (a micro-bench), this would flip to "extend" -- the existing `bench_harness` function with its `iter_custom` timing loop and the wired Perry/V8 arms would be the natural home. The brief says "publishes" and "artillery-style client," which firmly places it outside the in-process topology.

- **Perry-fork `benchmarks/honest_bench`:** classified "not applicable" because it benchmarks native executables, not SpacetimeDB modules. If the E2E harness needed to also include a standalone (non-STDB) Perry-vs-V8 comparison as a control arm, the `honest_bench` harness structure (warmup/measured phases, JSON output with metadata capture, correctness gate via output hashing) would flip to "extend" as a measurement-harness pattern to crib from. The brief does not mention standalone comparison, so it stays not applicable.

- **Pre-built `.wasm` artifacts:** classified "reuse" assuming the Perry compilation toolchain is already functional for `--target spacetimedb`. If the `empty.wasm` / `numk.wasm` artifacts need regeneration (the `artifacts/perry/` dir is currently empty on disk, though `modules.rs` references it), this depends on `perry compile --target spacetimedb` working at HEAD, which doc 16 confirms it does. The borderline is whether the existing artifacts are stale vs current HEAD -- if stale, it is still "reuse" (the path/pattern is reusable) but requires a rebuild step.
