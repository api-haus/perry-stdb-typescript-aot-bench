# 05 — perry-runtime core → wasm32-unknown-unknown port feasibility

This document answers the make-or-break risk for the sound path (doc 04 §2b, Option B): can perry-runtime's CORE — the GC, the NaN-boxed `JSValue`, and the Math/String/Array/Object/JSON/Number/BigInt builtins — compile to freestanding `wasm32-unknown-unknown`, the target a stock SpacetimeDB host loads? The short answer is yes, and the work is bounded. I reproduced the original `getrandom` failure, iterated the build to isolate every blocker, and drove the core to a state where it compiles with **zero errors in any core module** — the only 14 remaining compile errors are all in genuinely Node-shell modules (subprocess spawning, filesystem I/O, the in-process fuzzer) that a sandboxed SpacetimeDB module does not need. Every claim below is grounded at `file:line` in the local checkout `/mnt/archive4/DEV/mmodb/_vendor/perry` (branch `wasm-port-measurement`, off `main` at `4469d0f`) or against a build I ran. The full measurement diff is preserved at `/mnt/archive4/DEV/mmodb/_scratch/perry-wasm-port/measurement.diff` (325 lines, 9 files).

## 1. Core-versus-shell module map

perry-runtime is a single crate (`crates/perry-runtime`, ~120 source modules, `lib.rs` 506 lines pre-edit) built as one compilation unit producing `libperry_runtime.a` (`[lib] crate-type = ["rlib", "staticlib"]`, `Cargo.toml:8-9`). It is not internally partitioned into a portable core and a host shell — it is a monolithic Node-compatible runtime, and the core/shell boundary is something this analysis draws, not something the code declares. The `#[global_allocator]` is `mimalloc` (`lib.rs:24`), set crate-wide.

### CORE — value model, GC, JS builtins (must reach wasm32)

These modules implement the language itself and have no intrinsic dependency on fs/os/net. They compile cleanly to wasm32 once the dependency- and pointer-width-level fixes in §2 are applied (verified: §3).

- **Value model.** `value/` (4024 LOC across `jsvalue.rs`, `nanbox.rs`, `tags.rs`, `equality.rs`, `to_string.rs`, `truthy.rs`, `dynamic_*.rs`, `handle.rs`). `JSValue` is `#[repr(transparent)]` over a single `u64` (`value/jsvalue.rs:8-10`); NaN-boxing tags per the CLAUDE.md table. Pure bit manipulation, no OS surface.
- **Garbage collector.** `gc/` (33160 LOC across `mod.rs`, `malloc.rs`, `copying.rs`, `barrier.rs`, `oldgen.rs`, `policy.rs`, `cycle.rs`, `roots.rs`, `trace.rs`, `layout.rs`, `telemetry.rs`, `verify.rs`, `types.rs`, `roots/shadow_stack.rs`). Generational mark-sweep. The GC heap is allocated through `std::alloc` via `Layout::from_size_align` (`gc/malloc.rs:131,194,453`; `arena/block.rs:53,174`), which routes through the global allocator — **not** `mmap`/`VirtualAlloc` — so swapping the wasm allocator (§2) is the entire heap-portability story. Precise roots come from a TLS-backed shadow stack (`gc/roots/shadow_stack.rs`); the conservative stack scan uses `pthread_get_stackaddr_np` (macOS) / `pthread_getattr_np` (Linux) / TEB reads (Windows) and **already has a fallback `get_stack_bottom() -> 0` for every other target** (`gc/roots.rs:704-708`), so on wasm32 it degrades to "no conservative roots" and the shadow stack carries correctness.
- **Arena.** `arena/` (5054 LOC) — block allocator over `std::alloc`, same portability as the GC.
- **Builtins.** `builtins/` (6733 LOC: `arithmetic.rs`, `console.rs`, `numbers.rs`, `formatting*`, `globals.rs`, `table.rs`), plus `math.rs`, `json/` + `json_tape.rs`, `string/`, `array/`, `object/`, `bigint.rs`, `number`-handling in `builtins/numbers.rs`, `symbol.rs`, `set.rs`, `map.rs`, `collection_iter*`, `closure/`, `promise/`, `regex.rs` (Rust `regex`/`fancy-regex` — pure), `typedarray*`, `buffer/`, `text.rs`, `iterator_helpers.rs`, `proxy.rs`, `weakref.rs`, `disposable.rs`, `error.rs`, `exception.rs`, `frame.rs`.
- **`fs::validate` (misfiled core).** `fs/validate.rs` is named for `fs` but is a generic argument-validation / `TypeError`-builder module depending only on `crate::string` and `crate::value` (`fs/validate.rs:18-19`). It is called from 40 sites in `process.rs`, 17 in `dns.rs`, and from core `builtins/console.rs`, `builtins/formatting/strip_vt.rs`, `object/assert.rs`, `buffer/validate.rs`, etc. — 364 `crate::fs::` references originate outside `fs/`, dominated by `crate::fs::validate`. It is pure and must stay; its physical home under `fs/` is the single biggest obstacle to a clean module-level gate (see §5).

### SHELL — Node host surface (gate out for the SpacetimeDB target)

A SpacetimeDB module is sandboxed: no fs, os, net, crypto, subprocess, TTY. These modules are irrelevant to it and carry the wasm32 blockers.

- **Filesystem.** `fs/` minus `validate`/`errors`/`time` — `fs/mod.rs` (`std::fs`, `std::os::unix::fs::symlink` at `:1813`), `fs/fd_ops.rs` (`libc::c_char` at `:625`), `fs/cp.rs`, `fs/stats.rs`, `fs/stream.rs`, `fs/dirent.rs`, `fs/dir_glob_watch.rs`, `fs/filehandle.rs`.
- **Subprocess.** `child_process/` — `child_process/mod.rs` and `fork.rs` use `std::process::Command` + `libc`; `reactor.rs` uses `std::net`. The node `child_process` syscall router lives in core dispatch (`object/native_module_dispatch.rs:1484-1504`) and `lib.rs:359,392` (pump), `cluster.rs:108`, `util_promisify.rs:191` — this is the entanglement that blocks a trivial gate (§5).
- **OS / process / TTY.** `os.rs` + `os/` (libc, `hostname`, `dirs`; 61 external `crate::os::` refs), `process.rs` + `process/credentials.rs` (libc, `std::process`; 50 external refs, incl. `gc/mod.rs:109` and `gc/policy.rs:1182` calling `process::get_rss_bytes`), `tty.rs` (libc termios), `os_priority.rs`, `os_network.rs`.
- **Networking.** `net.rs` (not exported — `lib.rs:138`), `dns.rs`, `dgram.rs`, `tls.rs` (libc + `std::net`; 12 external `crate::tls::` refs incl. `gc/mod.rs:341`), `node_stream*` (14 files), `url/mod.rs`.
- **Threading / event loop.** `thread.rs`, `timer.rs`, `event_pump.rs` (libc `_exit`/`write`), `async_context.rs`, `async_hooks.rs`, `cluster.rs`, `perf_hooks.rs`.
- **Fuzzer / UI / platform.** `geisterhand_registry.rs` (libc `free`; registered in GC at `gc/mod.rs:346`), `tui/` (libc termios; budgeted GC scanners at `gc/mod.rs:356-364`), `ui_text_registry.rs`, `plugin.rs` (feature `full`), `node_v8.rs`, `i18n.rs`, the `*-game-loop`/`ohos-napi` cfg-gated modules.

### Dependency split

Default features pull (`Cargo.toml:44-70`, `cargo tree` at depth 1): `anyhow`, `base64`, `dirs`, `fancy-regex`, `hostname`, `idna`, `itoa`, `lazy_static`, `libc`, `mimalloc`, `rand`, `regex`, `ryu`, `serde_json`, `taffy`, `thiserror`, `unicode-normalization`, `url`. `--no-default-features` drops only `dirs` and `hostname` (the `full` feature, `Cargo.toml:19`). The wasm-relevant split:

- **Core-clean on wasm32 as-is:** `anyhow`, `base64`, `idna`, `itoa`, `lazy_static`, `regex`, `fancy-regex`, `ryu`, `serde_json`, `taffy` (`default-features = false`), `thiserror`, `unicode-normalization`, `url`. These are pure-Rust and compiled in the wasm build below without complaint.
- **Native-host only (must be retargeted):** `libc` (no freestanding wasm32 libc), `mimalloc` (`libmimalloc-sys` build.rs has no wasm path — `~/.cargo/.../libmimalloc-sys-0.1.49/build.rs`), `rand 0.8` → `rand_core 0.6` → **`getrandom 0.2.17`** (the originally-reported blocker).
- **Shell-only optional:** `dirs`, `hostname` (only `os.rs` uses them).

## 2. Ordered blocker → fix table (what I actually hit)

Iterating `cargo build -p perry-runtime --target wasm32-unknown-unknown --no-default-features` and fixing each blocker in turn:

| # | Blocker (where) | Why it fails on freestanding wasm32 | Standard fix | What I did |
|---|---|---|---|---|
| 1 | **`getrandom 0.2.17`** via `rand 0.8 → rand_core 0.6` (`compile_error!` at `getrandom-0.2.17/src/lib.rs:346`) | getrandom 0.2 hard-`compile_error!`s on `wasm*-unknown-unknown` unless a backend feature is set, because the target has no OS entropy syscall. | Enable getrandom's `custom` feature and `register_custom_getrandom!`; in a SpacetimeDB module the entropy source is the host (`ctx.random`), so a real port forwards to a host import. | `Cargo.toml`: `getrandom = { version = "0.2", features = ["custom"] }` under `[target.'cfg(target_arch = "wasm32")'.dependencies]`; `lib.rs`: `wasm_getrandom` module registering a deterministic zero-fill **stub** (clearly marked NOT secure — a real port wires `ctx.random`). |
| 2 | **`mimalloc`** `#[global_allocator]` (`lib.rs:24`) | `libmimalloc-sys` compiles a C library; its build.rs has no wasm32 branch, so the global allocator can't build. | Swap to a wasm-native allocator (`dlmalloc` or `wee_alloc`) under `cfg(target_arch="wasm32")`. | `Cargo.toml`: moved `mimalloc` + `libc` into `[target.'cfg(not(target_arch = "wasm32"))'.dependencies]`; added `dlmalloc = { version = "0.2", features = ["global"] }` for wasm32. `lib.rs:23-31`: cfg-split `GLOBAL` between `mimalloc::MiMalloc` and `dlmalloc::GlobalDlmalloc`. | 3 | **`1usize << 34` overflow** (`gc/barrier.rs:785,1046`, `gc/copying.rs:196`, `timer.rs:827`, `typedarray.rs:279`) | `usize` is **32-bit** on wasm32, so the 16 GB GC heap-size sanity bound `1usize << 34` overflows; `#[deny(arithmetic_overflow)]` makes it a hard error. This is the canonical pointer-width assumption that a 64-bit-only codebase bakes in. | Widen the constant to 64-bit: `(size as u64) > (1u64 << 34)`. | Rewrote all 5 sites. | 4 | **`libc::localtime_r`/`localtime_s`** in `date.rs` (`timestamp_to_local_components`, `#[cfg(unix)]`/`#[cfg(windows)]` at `:158,179`) | wasm32-unknown-unknown is neither `unix` nor `windows`, so the function vanished while 18 callers remained; the sandbox has no libc and no timezone database. | Add a `cfg(target_arch="wasm32")` arm returning UTC (`tz_offset = 0`) via the existing pure-Rust civil-calendar `timestamp_to_components`. | Added the wasm32 arm at `date.rs:158`; gated the unix arm to `all(unix, not(target_arch="wasm32"))`. `Date` is a JS builtin so this is core-adjacent; UTC-only is correct for a sandbox (host supplies the clock). |
| 5 | **GC init registers shell scanners** (`gc/mod.rs:307,341,342,343,346,347`) — `child_process::reactor`, `tls`, `os` (×2), `geisterhand_registry`, `ui_text_registry` | `js_gc_init` hard-wires root-scanner registration from shell modules, so the GC pulls the entire Node shell into its link closure. | Gate the shell-scanner registrations under `cfg(not(target_arch="wasm32"))`; a SpacetimeDB build has none of those live objects. | Wrapped the 6 registration calls. After this, the GC compiles with no shell dependency. (The budgeted `tui` scanners at `:356-364` and `plugin` at `:345` are already feature/`cfg`-gated.) |
| 6 (residual) | **fs I/O, child_process, geisterhand** — 14 errors: `child_process/mod.rs` (6: `std::process::Command` + libc), `geisterhand_registry.rs` (6: `libc::free`), `fs/mod.rs:1813` (`std::os::unix::fs::symlink`), `fs/fd_ops.rs:625` (`libc::c_char`) | Genuine host calls with no wasm32 equivalent: subprocess spawn, filesystem syscalls, raw libc free. | Gate these modules out of `lib.rs` for the SpacetimeDB target. Blocked from a clean gate by §5 (the node-module dispatch router and the `fs::validate` physical location). | Left as the measured residual — these are the SHELL, and the point was to prove the CORE compiles, not to finish the shell amputation. |

Blockers expected from the brief that **did not materialize** (negative space worth recording): `std::fs` at the *core* link level (the core never touches it — only the shell does); `dirs`/`hostname` (already optional, dropped by `--no-default-features`); `mio`/tokio sockets (the async socket path is in `perry-stdlib`, a *separate* crate, not perry-runtime); `malloc_trim` (`gc/cycle.rs:762` is already `#[cfg(target_env = "gnu")]`-gated, excluded on wasm32 automatically). `std::env::var` is used widely for `PERRY_*` debug toggles across core (gc, arena, json, promise) but **compiles fine** on wasm32 (it just always returns `Err` in the sandbox) — not a blocker.

## 3. Furthest-compiling state + the gating diff

**Final state: the core compiles with zero errors; 14 errors remain, all in shell modules.** Confirmed by filtering the build output for any error sited in a core module (`value|gc|builtins|json|math|string|array|object|bigint|arena|symbol|set|map|closure|promise|number`) — the result is empty. The 14 residual errors localize to exactly 4 shell files: `geisterhand_registry.rs` (6), `child_process/mod.rs` (6), `fs/mod.rs` (1), `fs/fd_ops.rs` (1).

The gating diff that achieved this (`/mnt/archive4/DEV/mmodb/_scratch/perry-wasm-port/measurement.diff`, also on branch `wasm-port-measurement`) touches 4 source files plus `Cargo.toml`/`Cargo.lock`:

- `Cargo.toml` — `libc`+`mimalloc` → `cfg(not(wasm32))`; `dlmalloc`+`getrandom/custom` → `cfg(wasm32)`.
- `lib.rs` — cfg-split `#[global_allocator]`; `wasm_getrandom` custom backend (stub).
- `gc/barrier.rs`, `gc/copying.rs`, `timer.rs`, `typedarray.rs` — 5× `1usize<<34` → `(… as u64) > (1u64<<34)`.
- `gc/mod.rs` — 6× `cfg(not(wasm32))` on shell root-scanner registrations.
- `date.rs` — wasm32 UTC arm for `timestamp_to_local_components`.

This is ~83 inserted lines. It is deliberately **not** a finished port: it is the minimum edit set that proves the core is wasm-portable and that the residual is confined to the shell.

When I additionally gated `child_process` and `geisterhand_registry` out of `lib.rs`, the error count did **not** drop to zero — it rose to a different 31 (`object/native_module_dispatch.rs:1484-1504`, `lib.rs:359,392`, `util_promisify.rs:191`, `cluster.rs:108` all reference `crate::child_process` directly). That negative result is the key §5 finding: the shell cannot be amputated by deleting module declarations, because the `require('<node-module>')` dispatch router in core lists every shell module by name. I reverted those two gates to keep the clean 14-error furthest state.

## 4. LLVM → wasm32 codegen + link plan

There is, confirmed, **no LLVM→wasm path today**. The two wasm-adjacent facts in the tree are unrelated to it:

- `perry-codegen-wasm` (439 LOC, `wasm-encoder`-based) is a **separate, hand-written bytecode emitter** that produces the `rt`-host browser artifact for `--target wasm` (= alias of `--target web`, doc 01/04). It does **not** depend on `perry-runtime` (`Cargo.toml` deps: `perry-hir`, `perry-types`, `perry-codegen-js`, `perry-dispatch`, `wasm-encoder`, `base64` — no perry-runtime). It imports the JS object model from a host (`wasm_runtime.js`). This is the artifact doc 04 ruled RED. It is irrelevant to the sound path.
- `perry-codegen` (the LLVM path) has **zero** wasm32 references (`rg wasm32|wasm-ld` in `perry-codegen/src` and `perry/src` → empty).

How Perry's LLVM codegen actually works, and what a wasm32 target entails:

- **Codegen is `.ll`-text + shell-out to `clang`, not inkwell/llvm-sys.** `perry-codegen/src/linker.rs` writes LLVM IR text to disk and runs `clang -c -O3` to produce a `.o` (`linker.rs:1-7,67,107,203`), then the `perry` driver links the `.o` set with `cc`/`lld` (`crates/perry/src/commands/compile/link/`). The module string carries `target triple = "<triple>"` (`perry-codegen/src/module.rs:221`) but **no explicit `target datalayout`** — so clang derives the wasm32 datalayout from the triple. This is the seam that makes a wasm32 target tractable: it is mostly a triple + linker swap, not an IR rewrite.
- **Adding the target.** `resolve_target_triple` (`perry-codegen/src/codegen/helpers.rs:299-319`) maps `--target <name>` → LLVM triple; it has 16 entries (ios, android, linux, …) and no wasm. A `"spacetimedb"` / `"wasm32"` entry returning `"wasm32-unknown-unknown"` is a one-line addition. `default_target_triple` (`:265-279`) is host-only and untouched.
- **Linker.** The driver's link step (`link/platform_cmd.rs`, `link/mod.rs`) selects `clang`/`cc`/`ld64.lld`/`lld-link` per platform. wasm32 needs a new arm shelling to **`wasm-ld`** (present at `/usr/bin/wasm-ld`; clang 22 lists `wasm32`/`wasm64` targets — verified locally) with reactor-style flags: `--no-entry`, `--export=__describe_module__`, `--export=__call_reducer__`, `--export=memory`, `--import-memory` or default, `--allow-undefined` only for the `spacetime_10.x` host imports. The `clang -c` step needs `--target=wasm32-unknown-unknown` instead of the host `-march=native` tuning (`linker.rs:42-48`).
- **Entry shape.** Codegen currently emits a `main` (`perry-codegen/src/codegen/entry.rs:134`, `define_function("main", I32, …)`). The STDB native contract is a **reactor** (no `_start`/`main`; host calls `__describe_module__`/`__call_reducer__` exports on demand, doc 02/04). A wasm32 codegen mode must suppress `main`, emit the two dunder exports with the exact STDB signatures (`__describe_module__: (i32)->()`, `__call_reducer__: (i32 i64×7 i32 i32)->i32`, doc 04 §1), and import ≥1 `spacetime_10.x` host func for ABI detection — none of which the current emitter can name or sign (doc 04 §1 walls). This is the new-backend work, the same multi-week item doc 04 §2b called Option B; the value of this doc is that it shows the *runtime half* of that backend is far cheaper than the codegen half.
- **Is the nanbox value model wasm-compatible as-is?** Two parts:
  1. **Pointer width: yes, comfortably.** Nanbox stores pointers in the low 48 bits of the f64 (`value/nanbox.rs:223,278`, `POINTER_MASK`). wasm32 pointers are 32-bit (≤ 4 GB ≪ 2⁴⁸), so they fit with room to spare — no value-model breakage. This is a genuine positive: the representation does not have to change.
  2. **NaN-canonicalization: a real codegen-correctness constraint, not a wall.** `JSValue` is a `u64` (`value/jsvalue.rs:8-10`) but at the LLVM level Perry carries values as `double` — function params/returns are `DOUBLE`, locals are `alloca(DOUBLE)` + `store/load DOUBLE` (`perry-codegen/src/codegen/function.rs:51,56,98-99,305`; runtime decls pass `DOUBLE`, `runtime_decls/strings.rs:19,28-29`). On x86/arm, moving a NaN-boxed double through registers/memory is bit-preserving. The wasm spec **permits NaN canonicalization for f64 *arithmetic* ops** (`f64.add/mul/neg/abs/copysign/...`) but **not** for `local.get/set`, `global.get/set`, `f64.load/store`, or `call` argument passing, which are bit-preserving. So the model is wasm-safe *provided* LLVM never routes a NaN-boxed value through a float-arithmetic instruction during pure data movement — which it should not (it only does arithmetic on values it has proven are real numbers). This must be verified empirically on the first compiled kernel (dump the `.wasm`, confirm no `f64.neg`/`f64.copysign` appears on tag-carrying values), and is the single codegen risk to flag.

## 5. Effort estimate — "core + LLVM-wasm32 codegen → freestanding wasm module"

Sub-tasks, each with a risk flag. The runtime-port half is now de-risked (this doc); the codegen-backend half remains the bulk and inherits doc 04's "Option B is multi-week" verdict.

**A. perry-runtime core → wasm32 (de-risked here).**
- A1. Land the §2 dependency/pointer-width fixes (getrandom custom backend, dlmalloc, 5× u64 widening, date UTC arm, GC shell-scanner gates). **~1 day.** Risk: LOW — done and building in the measurement branch; just needs the stub RNG replaced by a real host import and review.
- A2. Amputate the shell cleanly. Extract `fs::validate` + `fs::errors` + `fs::time` out of `fs/` into a top-level pure module (e.g. `argval`/`jserr`) so the fs I/O can be gated without taking the validators with it; introduce a `stdb-core` cargo feature (or `cfg(target_arch="wasm32")`) that excludes `fs` I/O, `child_process`, `os`, `process`, `net`/`dns`/`dgram`/`tls`, `tty`, `thread`, `timer`/`event_pump`, `geisterhand_registry`, `tui`, `node_stream*`, `perf_hooks`, `cluster`; **and trim `object/native_module_dispatch.rs` to the JS-builtin subset** so the `require('fs'|'child_process'|…)` router stops hard-referencing the gated modules (`:1484-1504`). **~3–5 days.** Risk: MEDIUM — the dispatch router is the real entanglement (§3 negative result); the validator extraction touches ~360 call sites' import paths (mechanical but broad).
- A3. Host-import shims for the few core needs a sandbox still has: RNG (`Math.random`, `crypto.getRandomValues` → host `ctx.random`), time (`Date.now` → host timestamp), `console.log` → STDB `console_log`. **~1–2 days.** Risk: LOW–MEDIUM — small surface, but each is a new wasm import that must match a host-provided signature.

**B. LLVM → wasm32 codegen backend (the bulk; = doc 04 Option B).**
- B1. Target plumbing: `resolve_target_triple` entry + `clang --target=wasm32-unknown-unknown` compile arm + `wasm-ld --no-entry` link arm with the reactor export flags. **~2–3 days.** Risk: LOW–MEDIUM — well-trodden LLVM/clang/wasm-ld path; the seam (`.ll`→clang→`.o`→ld) already exists.
- B2. Reactor entry + ABI-exact dunders: suppress `main`; emit `__describe_module__`/`__call_reducer__` with the exact STDB signatures and the `spacetime_10.x` import(s). This requires extending codegen to (a) pin chosen export *names*, (b) pin chosen *signatures* (mixed i32/i64, not the forced NaN-box ABI), (c) declare imports from a chosen namespace — the three capabilities doc 04 §1 proved Perry lacks today. **~1.5–3 weeks.** Risk: HIGH — this is new compiler surface, not a flag; it is the load-bearing unknown of the whole project.
- B3. Addressable linear memory for BSATN: a TS-level primitive (or codegen intrinsic) to read/write bytes at a chosen wasm linear-memory offset, so the module can marshal `bytes_source_read`/`bytes_sink_write` (doc 04 §1 wall #5). **~1–2 weeks.** Risk: HIGH — Perry's TS surface has no addressable-memory concept today; this is design work, not just codegen.
- B4. NaN-canonicalization verification + fix: dump the first compiled wasm, confirm tag-carrying f64 values never hit a canonicalizing op; if they do, force i64 transit for value moves on the wasm backend. **~2–4 days.** Risk: MEDIUM — the §4 analysis says it *should* be safe, but "should" on a NaN-box-through-f64 model on wasm is exactly the kind of thing that bites and must be measured, not assumed.
- B5. Integration + benchmark: compile the doc-04 `mix` kernel as a real STDB module, publish to the pinned 2.0.1 server, run it on the stock Wasmtime host alongside the V8 arm. **~3–5 days.** Risk: MEDIUM — first real end-to-end; gated on B2/B3 landing.

**Rolled up:** the **runtime core** is **days** of bounded work (A1–A3 ≈ 1–1.5 weeks, mostly de-risked). The **codegen backend** is **multi-week** (B ≈ 4–7 weeks), dominated by B2/B3, both HIGH risk and both genuinely new compiler capability. Net for "freestanding wasm module that satisfies the STDB ABI": **multi-week**, with the critical path entirely in codegen (B2/B3), not in the runtime. The brief's make-or-break risk — "can the core reach wasm32" — resolves **green**: it can, with a small, measured patch set.

## Side notes / observations / complaints

- **The brief's framing was right and the original probe was a paper cut, not a wall.** `getrandom` was one `Cargo.toml` line away from resolved; the real surface is `libc` + `mimalloc` + a 32-bit-`usize` overflow, all standard wasm-port friction. Nothing in the *core* is architecturally hostile to wasm32. The GC routing through `std::alloc` (not `mmap`) and the pre-existing `get_stack_bottom()->0` fallback are the two facts that make the GC port cheap — whoever wrote those left the door open, probably without intending a wasm target.
- **The core/shell boundary is real but undeclared, and the dispatch router is where it leaks.** The single biggest obstacle to a clean `cfg`-gate is not any individual blocker — it is `object/native_module_dispatch.rs`, the `require('<node-module>')` switch, hard-listing every shell module by name (`:1484-1504` for child_process alone). Gating modules out of `lib.rs` without trimming this router just relocates the errors. A `stdb-core` build needs the dispatch table cut to the JS-builtin subset, which is the bulk of A2's risk.
- **`fs::validate` is misfiled and it costs.** A pure argument-validator / `TypeError`-builder used by 360+ core call sites lives physically under `fs/`. It compiles fine, but it means you cannot gate `fs` out by module declaration without first relocating its validators. This is exactly the kind of "abstraction fighting the directory layout" that turns a one-line gate into a 360-call-site import churn. If perry ever wants a portable core, extracting this is the first refactor.
