# 13 — Context bundle: the first real Perry-vs-V8 number (synthetic, de-risk)

Canonical context for the phase that produces the **first real Perry-vs-V8 performance number**. Every non-review agent in this phase reads this file first, then its group file. Self-contained: code refs are `file:line` at current HEAD, not paraphrases.

## Behavioural problemspace

**How it behaves now.** The V8 (control) arm produces real numbers through SpacetimeDB's own `crates/bench` suite: `empty` reducer 7.20µs (≈ native Rust 7.14µs — dispatch is at parity), and the whole `special` CPU/marshaling/game suite (`print_bulk` 1.32–4.87×, `circles` 2.27–2.39×, `ia_loop` 0.97–1.98× vs Rust). The **Perry** arm produces **zero** numbers: a Perry-compiled module loads on a stock SpacetimeDB host and its reducer is *callable*, but `__call_reducer__` is a **no-op** — it `(void)`-casts every parameter and `return 0;` (`_vendor/perry-fork/crates/perry/src/commands/compile/spacetimedb.rs:114-122`). The user's reducer body never runs. The describe blob is real and server-decoded but hardcoded for a single reducer named `noop` (`spacetimedb.rs:85-99`, the name at `:92`).

**How it is supposed to behave.** The same `benchmarks-ts`-style source compiled by Perry into a freestanding `wasm32` module (`host_type=Wasm`), loaded by the *stock* Wasmtime host exactly like a Rust/C# module, runs its reducer body for real, so `cargo bench` measures a genuine Perry-AOT-vs-V8-JIT comparison on one Criterion clock (canon: `10-benchmark-landscape.md`). Behavioural target for this phase: an `empty`-reducer Perry number to compare against V8's 7.20µs, and a **pure-integer CPU kernel** number (the AOT-vs-JIT discriminator — V8 was ~2.3× slower than native on the compute-heavy `circles`/`ia_loop` rows, which is where Perry's thesis lives).

**What the user wants.** Verbatim: *"i want Perry vs V8 benchmark numbers on STDB synthetic benchmarks."* And the sequencing decision (AskUserQuestion, 2026-06-01): **de-risk synthetic first** — land this isolated CPU-kernel number to prove AOT actually beats V8-JIT *before* investing in the demos + docker e2e harness + M3 datastore work. Success = a trustworthy first Perry-vs-V8 number, adversarially verified to be the *real dispatched body* and not an artifact.

## Restated goal

Turn Perry's no-op `__call_reducer__` into a real dispatcher (the single audited gap = "M2-proper"), then produce two apples-to-apples Perry-vs-V8 numbers on `crates/bench`'s Criterion clock: (1) `empty` (call overhead), (2) a pure-integer CPU kernel (xorshift/mix — no datastore, no runtime archive). No demos, no docker, no M3 datastore in this phase.

- **Sequencing (AskUserQuestion 2026-06-01): "De-risk synthetic first."** Land M2-proper (empty + CPU integer kernel) → first real Perry-vs-V8 number, before demos/harness/M3.
- **First demo workload (AskUserQuestion 2026-06-01): "inventory+stats MMORPG."** This is the *first demo to build LATER*, in the subsequent (deferred) demo phase — it does NOT drive this phase. The CPU kernel here is a pure-integer xorshift/mix, NOT inventory+stats. Recorded so no agent conflates the two.
- **CLI edits are permissible for the eventual upstream PR** (user, verbatim handoff): *"things like CLI edits are permissible — its the PerryWasm engine pathway what I rejected since its obviously bad architectural idea."* So the `--engine perry` CLI path is a legitimate eventual product surface — but the goal *now* is the number, which argues for the smallest bench wiring first (see Open questions). The rejected `PerryWasm` host-fork approach is off the table permanently (memory `no-perrywasm-host-fork`).
- **Measurement surfaces:** for the eventual demos, BOTH an e2e docker-compose harness AND an isolating in-process micro-bench (the orchestrator committed to both; e2e alone dilutes Perry's reducer-only contribution under network/persistence). For *this* synthetic CPU-kernel phase, the surface is the in-process `crates/bench` Criterion micro-bench only — a pure-CPU kernel needs no docker/e2e.

## Reuse audit summary (relayed facts — full detail in `12-bench-audit-{stdb,perry}.md`)

**The Perry-side gap is exactly one thing** (`12-bench-audit-perry.md`): `__call_reducer__` no-op (`spacetimedb.rs:114-122`). Everything else for a benchmarkable module already works and is server-verified: the wasm32 triple (`perry-codegen/src/codegen/helpers.rs:310`), the reactor shape (`entry.rs:84-85`), named typed dunders + the one `spacetime_10.0` import (`spacetimedb.rs:82-83,102,114,128,287-289`), a real 34-byte `RawModuleDef::V10` describe blob (`spacetimedb.rs:85-99`, server-decoded per docs 09/09b), integer arithmetic codegen (M1 §3: `add`→`i64.add`). The shim is a C source string `STDB_ABI_SHIM_C` (`spacetimedb.rs:73-123`) compiled by Perry's own clang at link time (`compile_abi_shim`, `:133-176`) and linked with `wasm-ld --no-entry --gc-sections --export=… ` and **no `--allow-undefined`** (`:287-295`) — an undefined symbol is the *designed* signal that the runtime archive must be linked. User `perry_fn_*` symbols are already discovered (`collect_user_function_exports`, `:210-242`).

**The STDB-side wiring is a thin extension of three existing seams** (`12-bench-audit-stdb.md`): the `ModuleLanguage` marker trait + impls (`crates/testing/src/modules.rs:343-390`), the `spacetimedb_cli::build` language→`host_type` tag (`crates/cli/src/tasks/mod.rs:30-34`), and the `host_type`-driven native-vs-V8 load path (already exercised by Rust/C#). **No host change, no new `HostType` variant.** Critical correction: `host_type` is decided by **language detection, not magic-byte sniffing** (`mod.rs:30-34`: JavaScript→`"Js"`, else→`"Wasm"`) — a Perry build that still detects as JavaScript would be tagged `"Js"` and run under V8 *even with wasm bytes*. The Perry arm MUST force the `"Wasm"` tag. The Criterion group for a Perry arm would be `stdb_module/typescript-perry`. Killer probe / narrowing filter: `cargo bench -p spacetimedb-bench --bench generic -- "stdb_module/typescript/…"` (Criterion's trailing positional is a substring filter). The bench is fail-fast: a failed arm `.unwrap()`/`.expect()` panics the whole binary (`generic.rs:32-44`, `modules.rs:170`) — the three TEMP-LOCAL C# comment-outs in the working tree exist for exactly this reason and must stay for any run.

## Open questions / unresolved forks (resolve from code + canon — NOT pre-decided)

Quoted/condensed from the two audits' `## Borderline calls`. The architect resolves these from code + the relayed user constraints; they are NOT settled here.

- **Which Perry-build wiring shape?** (a) pre-built-artifact bypass — `TypeScriptPerry::get_module()` points at a Perry-produced `.wasm` on disk and hardcodes `host_type=HostType::Wasm`, zero CLI changes, number soonest; (b) a new `ModuleLanguage` enum arm in the CLI routing to a Perry builder; (c) an `--engine perry` parameter threaded into `build`/`build_javascript`. Audit flip condition: the bypass is insufficient if the bench must exercise the *real* Perry CLI pipeline (so the benchmark also validates the build path). User constraint: goal is the *number now* (de-risk), and CLI edits are permissible *later* — weigh accordingly, but it is the architect's call to ground from the code.
- **Does generating the describe blob from the user's schema land on the critical path for the first number, or can dispatch ship against a still-hardcoded def** named for the kernel (one-line edit of `spacetimedb.rs:92`), deferring the general schema walk to M3? Flip: whether the bench harness requires the reducer to be named/shaped exactly as the V8 arm's reducer, or tolerates a fixed-name stand-in.
- **Does the chosen CPU kernel genuinely stay shell-amputated, or does some TS construct pull in `js_*`?** Only answerable by compiling the actual kernel TS with `--target spacetimedb` and reading `wasm-ld` output: clean link = amputated; `undefined symbol: js_*` = the *unbuilt* wasm32 runtime archive (15 shell errors, per docs 05/08) is suddenly on the critical path — a circuit-breaker that reorders the milestone. Pick a kernel most likely to stay amputated (pure integer arithmetic, no arrays/strings/allocation/objects — like M1's proven `add`).
- **Is "M2-proper" one milestone or two?** empty-dispatch and CPU-kernel collapse into one change if the kernel takes no args (constants baked in, stays amputated); they split if the kernel takes args (arg-decode = declaring `bytes_source_read` + a minimal BSATN integer decoder is its own chunk).
- **Does a single-reducer `#[test]` exist** (`load_module` + one `call_reducer`) that would be a cheaper killer probe than the bench filter? Not exhaustively searched.

## Constraints (solution constraints with hard provenance — zero exceptions)

- **Never modify the SpacetimeDB host/core.** The Perry module must load on the *stock* Wasmtime host. Provenance: the project's committed architecture (memory `mmodb-project`; the rejected host-fork is memory `no-perrywasm-host-fork`). The audit confirms no host change is needed.
- **The Perry bench arm must force `host_type="Wasm"`.** Provenance: code fact `crates/cli/src/tasks/mod.rs:30-34` — host_type is language-tagged, not byte-inferred; a JavaScript-detected Perry build would silently run under V8. (This constrains the wiring shape, whichever is chosen.)

## Success criteria

- A Perry-compiled module whose `empty` reducer **genuinely dispatches and runs the user body**, benchmarked against V8's `empty` (7.20µs) on the same Criterion clock.
- A **pure-integer CPU kernel** reducer (xorshift/mix) defined ONCE and run on BOTH the V8 arm and the Perry arm (apples-to-apples), on the same clock — the AOT-vs-JIT discriminator. (The stock suite has no pure-CPU integer kernel — this workload is added, per `10-…md` §28.)
- **Adversarially verified by fresh eyes** (an agent that did not write the code) that the measured Perry cost is the *real dispatched body*: `wasm-tools print`/disasm shows `__call_reducer__` actually `call`s `perry_fn_*` and that function survived `--gc-sections`, AND a side-effect or non-trivial-duration assertion confirms the kernel ran. A "fast" number that is actually a GC'd-away empty call is the central failure mode (both audits + negative-space pt 6) and must be ruled out before any number is reported as real.
