# 06 — STDB TypeScript-module fork surface and the "release-compile with Perry" option

This document designs exactly how a fork of the SpacetimeDB TypeScript module (npm `spacetimedb`, source `crates/bindings-typescript`) gains one user-facing option: compile the same TS module with Perry for release builds, while development builds keep the normal rolldown→V8 path. It locates the single branch point in the CLI where the option lives, audits every `src/server` file for Perry-compilability, extracts the C#/C++ ABI-lowering pattern the Perry path must mirror, specifies the embedding, and states the dev/release equivalence contract the PoC validates.

Scope boundary inherited from the locked decisions: the SpacetimeDB host and module loader are never modified. A Perry-compiled release build publishes with `host_type=Wasm` and is loaded by the stock Wasmtime loader exactly like a Rust/C#/C++ module. Reaching that contract is a new Perry codegen backend (`--target spacetimedb`), not a flag on the existing `--target wasm`; doc 04 established this is RED today and the work is the Perry backend, and this document does not relitigate that — it designs the surface that *consumes* such a backend once it exists, and is explicit about which parts are buildable now versus blocked on the backend.

Every load-bearing claim below is grounded at `file:line` in `/mnt/archive4/DEV/mmodb/_vendor/SpacetimeDB` or `/mnt/archive4/DEV/mmodb/_vendor/perry`, or in the ABI contract verified in `02-stdb-abi.md`.

## 1. The release-option hook — where the branch lives

The CLI build flow funnels every language through one function that returns both the artifact path and the host_type string. `tasks::build` in `crates/cli/src/tasks/mod.rs:13-58` calls `util::detect_module_language` (`tasks/mod.rs:19`), dispatches to the per-language builder (`tasks/mod.rs:23-28`), and then assigns the host_type: a JavaScript project unconditionally returns `(output_path, "Js")` at `tasks/mod.rs:30-31`, every other language returns `(wasm_path, "Wasm")` at `tasks/mod.rs:33,56`. That returned `&'static str` is the host_type, and it flows untouched: `subcommands/build.rs:88` returns the tuple from `run_build`; `subcommands/publish.rs:561-568` receives it as `(path_to_program, host_type)`, reads the artifact bytes (`publish.rs:569`), and forwards the string verbatim as the publish query parameter at `publish.rs:637` (`builder.query(&[("host_type", host_type)])`).

The server consumes that query parameter as the `HostType` enum (`crates/core/src/messages/control_db.rs:96-100`: `Wasm = 0`, `Js = 1`) and routes on it with no other discriminator: `host_controller.rs:717-735` sends `HostType::Wasm` to `runtimes.wasmtime.make_actor` (`:721`) and `HostType::Js` to `runtimes.v8.make_actor` (`:730`). A module published as `host_type=Wasm` therefore lands in the stock Wasmtime loader regardless of the source language it was compiled from. This is precisely why a Perry-compiled freestanding wasm needs no host change: it is, to the server, an ordinary Wasm module.

**The branch belongs in `tasks/mod.rs`, not in `javascript.rs` and not in the publish path.** Today the language→host_type mapping is a fixed table at `tasks/mod.rs:30-34`. The release option splits the `ModuleLanguage::Javascript` arm into two: the dev path (rolldown→`dist/bundle.js`, host_type `"Js"`, the current `build_javascript`) and the release path (Perry→`dist/module.wasm`, host_type `"Wasm"`, a new `build_javascript_perry`). The host_type string the arm returns is what makes the published artifact a Wasm module; nothing downstream needs to know Perry exists.

### The flag surface — `spacetime build` gains one flag, publish inherits it for free

The `build` subcommand declares its own clap arguments in `subcommands/build.rs:8-42` (`module_path`, `lint_dir`, `features`, `debug`). The release option is one new flag here, e.g. `--engine perry` (a value-enum defaulting to the existing rolldown engine) or, more in keeping with the existing `--debug` flag, a `--release-engine perry` opt-in. The flag is read in `build::exec` (`build.rs:44-65`) alongside `build_debug = args.get_flag("debug")` (`build.rs:61`) and threaded into `run_build` → `tasks::build`, whose signature (`tasks/mod.rs:13-18`) gains one parameter (the chosen engine) mirroring the existing `build_debug: bool`.

Crucially, `spacetime publish` needs no new argument. Publish already forwards an arbitrary build-options string into the build subcommand: `publish.rs:514-515` reads the `--build-options` arg (defined `publish.rs:198-205`, default `""`), and `publish.rs:561-567` passes it to `build::exec_with_argstring`, which re-parses it through the full `build::cli()` parser (`build.rs:94-115`, `exec_with_argstring_argv` at `:117-125`). So `spacetime publish --build-options="--engine perry"` reaches the new flag with zero publish-side changes. The existing `--bin-path`/`--js-path` escape hatches (`publish.rs:213-232`) that publish a pre-built artifact directly already conflict-with `build_options` and remain available for hand-built Perry wasm during PoC bring-up.

### Why not a config-file or package.json field

The "one option" can be surfaced three ways; the CLI flag is the right one for the negative-space reasons below, and the other two are not mutually exclusive but are not the primary hook.

- **CLI flag on `build` (recommended).** Lives at the existing decision point (`build.rs` args → `tasks/mod.rs` branch), is per-invocation (so the same checkout builds dev or release without editing tracked files), and is reachable from publish through the existing `--build-options` channel with no new plumbing. It mirrors `--debug` exactly.
- **A field in `spacetime.json` / package.json.** `detect_module_language` already reads `package.json`'s existence (`util.rs:270`) and there is a standing TODO to add a config file recording the language (`util.rs:251`). A `"perryRelease": true` field could set the default engine. This is a reasonable *default-source* but a poor *primary switch*: it makes "dev vs release" a tracked-file edit, which fights the whole point (fast dev iteration on the same source). Use it only to set the project-level default that the flag overrides.
- **A tsconfig-style setting.** Wrong layer: tsconfig governs type-checking and the `tsc --noEmit` pass (`javascript.rs:47-65`), not artifact lowering. Perry has its own SWC-based front-end; coupling the engine choice to tsconfig conflates two compilers.

The flag is the canonical home; the config field is an optional default feeding it.

## 2. Target semantics the Perry path must replicate

The Perry-compiled wasm must be observationally identical to the V8 module on the two host-visible entry points, `__describe_module__` and `__call_reducer__`. The subtlety is that **the TS `ModuleHooks` interface is not the wasm ABI** — it is a JS-shaped interface that the Rust V8 host adapts to the wasm-level ABI. The Perry path has no such adapter; it must emit the wasm-level dunders directly. The two layers:

The JS hooks (`crates/bindings-typescript/src/server/sys.d.ts:15-47`) are JS-typed: `__describe_module__(): Uint8Array` returns a JS byte array; `__call_reducer__(reducerId: u32, sender: u256, connId: u128, timestamp: bigint, argsBuf: DataView): void` receives a JS `DataView` of already-extracted argument bytes and a `bigint` timestamp. The V8 host does the wasm-level marshalling on the Rust side: `crates/core/src/host/v8/syscall/v2.rs:441-470` (`call_call_reducer`) serializes `reducer_id`/`sender`/`conn_id`/`timestamp` into JS values, copies the BSATN args out of the host BytesSource into a JS `ArrayBuffer`, wraps it as a `DataView` (`reducer_args_to_value`, `v2.rs:494-521`), and invokes the JS hook. For describe, `v8/syscall/common.rs:126-139` calls the JS `__describe_module__`, takes the returned byte array, and the host writes it to the description sink.

The wasm-level ABI the stock loader actually requires (verified in `02-stdb-abi.md` §i): export `__describe_module__` as `(param i32) -> ()` taking a `BytesSink` id and writing BSATN to it via `bytes_sink_write`; export `__call_reducer__` as `(param i32 i64 i64 i64 i64 i64 i64 i64 i32 i32) -> i32` where the seven i64s are sender[0..3]/conn[0..1]/timestamp and the trailing two i32s are the args `BytesSource` id and the error `BytesSink` id; import host functions only from `spacetime_10.0..10.5`; export `memory`.

So the Perry-compiled module must **fold the V8 host's Rust-side marshalling into its own wasm**. Concretely it must replicate the following pieces of `src/server/runtime.ts`, lowered onto the `spacetime_10.x` imports rather than onto the `spacetime:sys` V8 externs:

- **Module-def construction (describe).** `ModuleHooksImpl.__describe_module__` (`runtime.ts:377-384`) builds the BSATN with `RawModuleDef.serialize(writer, RawModuleDef.V10(this.#schema.rawModuleDefV10()))` into a `BinaryWriter`. The byte layout this produces is the V10 layout verified in `02-stdb-abi.md` §iv (e.g. the 34-byte minimal def). In the Perry path the same serialization runs, but the resulting bytes are pushed to the `BytesSink` id via `spacetime_10.0::bytes_sink_write` in a drain loop, instead of being returned as a `Uint8Array` for the host to push.
- **Reducer registration / schema → module def.** The whole registration pipeline (`reducers.ts:55-111` `registerReducer`, `schema.ts:187-205` `[moduleHooks]`, `schema.ts:615-650` `schema()`) is plain TS object/array manipulation building `ctx.moduleDef`. It runs once at module init to produce the schema object that describe serializes. Note `reducers.ts:83-84` hardcodes `okReturnType = Product([])` (reducers return no value) and `errReturnType = String` — the Perry path inherits this identical module-def shape, so describe output is byte-identical across paths.
- **Arg decode + dispatch (call_reducer).** `ModuleHooksImpl.__call_reducer__` (`runtime.ts:394-414`) selects the per-reducer deserializer precomputed in the constructor (`runtime.ts:352-354`: `ProductType.makeDeserializer(params, typespace)` per reducer), resets a shared `BinaryReader` over the args buffer, decodes the arg product, resets a cached `ReducerCtx`, and calls `moduleCtx.reducers[reducerId]`. In the Perry path the args buffer is not handed in as a `DataView`; the module must first pull the BSATN bytes from the args `BytesSource` id via `bytes_source_remaining_length` + `bytes_source_read` (the exact loop the C++ `ConsumeBytes` runs, see §3), then run the same `BinaryReader`/`ProductType` decode, then dispatch by id. On user-thrown error it writes a UTF-8 message to the error `BytesSink` and returns `HOST_CALL_FAILURE`; on success returns `0`.

### Per-file Perry-compilability audit of `src/server`

The question for each file is: does it depend on the V8 host bridge (the `spacetime:sys` externs declared in `sys.d.ts`, or V8-specific intrinsics), and if so, can that dependency be re-pointed at the `spacetime_10.x` wasm imports, or does it need a Perry-compatible reimplementation? "Compilable" here means "Perry's TS surface can lower it once `--target spacetimedb` can emit the imports" — it does not mean Perry compiles it today.

- **`db_view.ts` (21 lines).** Pure type declarations (`ReadonlyDbView`, `DbView` mapped types). No runtime, no host dependency. **Compilable as-is** (types are erased).
- **`reducers.ts` (113 lines).** Registration logic operating on plain objects/arrays/`Map`/`Set` and the `AlgebraicType`/`RowBuilder` helpers. The one host-adjacent line is `Object.defineProperty(fn, 'name', ...)` (`reducers.ts:107`) — ordinary JS reflection within Perry's documented surface. **Compilable**, no reimplementation; depends only on Perry supporting `Map`/`Set`/closures (it does, per `00-context.md`).
- **`schema.ts` (650 lines).** Class-heavy (`SchemaInner`, `Schema`) with symbol-keyed methods (`moduleHooks`, `registerExport`, `exportContext`), `Map`/`Set`, `Object.entries`/`fromEntries`. All within Perry's class/generic/closure surface. The only host coupling is importing `moduleHooks` from `spacetime:sys@2.0` (`schema.ts:1`) — a `unique symbol` used as a method key, not a host call. **Compilable**; the `moduleHooks` symbol becomes an ordinary in-module symbol since there is no V8 host to call it (the Perry entry shim invokes `[moduleHooks]` itself — see §5).
- **`console.ts` (81 lines).** Every method routes to `sys.console_log` / `sys.console_timer_start` / `sys.console_timer_end` (`console.ts:23-75`), i.e. the `spacetime:sys` externs. Plus it imports `object-inspect` for formatting (`console.ts:3`). **Needs re-pointing, not reimplementation:** the `sys.*` calls must bind to `spacetime_10.0::console_log` / `console_timer_start` / `console_timer_end` (signatures in `02-stdb-abi.md` §ii). `object-inspect` is a third-party JS dependency whose Perry-compilability must be confirmed (it is plain JS string formatting; likely compilable, but it is the one external import in this file).
- **`polyfills.ts` (4 lines).** Imports `url-polyfill` and installs the custom `console` onto `globalThis` (`polyfills.ts:1-4`). `globalThis.console = console` is fine. `url-polyfill` is only needed by code paths that use `URL` (HTTP handlers); a CPU-bound reducer module does not, so for the PoC this import is **trimmable**. In general it **needs reimplementation-or-trim** depending on whether the user module touches `URL`.
- **`runtime.ts` (1102 lines).** The host-bridge core. It imports both `spacetime:sys@2.0` and `@2.1` namespaces (`runtime.ts:1-2`) and merges them into `sys` (`runtime.ts:62`); every datastore/console/identity/timer call goes through `sys.*`. The `ModuleHooksImpl` describe/call_reducer methods are the target semantics above. **Split verdict:**
  - The BSATN serialization/deserialization, `BinaryWriter`/`BinaryReader`, `ReducerCtx`, schema-walking, and the table-view builder (`makeTableView`, `runtime.ts:562-988`) are plain TS — **compilable**, re-pointing each `sys.*` call to the matching `spacetime_10.x` import.
  - The wasm-ABI boundary that the V8 host currently provides — pulling args from a `BytesSource` id, pushing describe/error bytes to a `BytesSink` id, and *exposing the two dunders with the exact wasm names and signatures* — **does not exist in this file at all** and must be **newly written** in the Perry path. This is the `bytes_source_read`/`bytes_sink_write` drain loops plus the export shim. It is the part that depends on the new Perry backend's ability to (a) emit `import_name`/`import_module` imports and (b) export functions with chosen names and the exact mixed-i32/i64 signature — the two capabilities doc 04 proved Perry's `--target wasm` lacks.
  - V8-specific runtime features used: `FinalizationRegistry` (`runtime.ts:1071`) and `Symbol.dispose`/`using` (`runtime.ts:994,1096`) for iterator cleanup, and private class fields (`#schema` etc.). These are only on the datastore/iterator paths; a CPU-bound reducer never reaches them. For the PoC's reducer-only module they are **trimmable**; for a full datastore-touching module they require Perry support for `FinalizationRegistry` and explicit-resource-management, which must be confirmed against Perry's surface (not in the documented-supported list in `00-context.md`).
- **`sys.d.ts` (125 lines).** Pure ambient `declare module 'spacetime:sys@2.0'` / `@2.1` — the V8 host externs. **This file is the thing that must be replaced**, not compiled: it declares the JS-shaped host interface (e.g. `datastore_insert_bsatn(table_id, row: ArrayBuffer, row_len): u32`, `console_log(level, message: string)`). The Perry path needs a parallel declaration that maps each of these to a `spacetime_10.x` wasm import with the *wasm* signature (pointer+length i32 pairs, not `ArrayBuffer`/`string`), exactly as the C++ `abi.h` does (§3). The string/ArrayBuffer-shaped V8 signatures here (e.g. `console_log` takes a JS `string`) hide a marshalling step the V8 host performs; the Perry path must marshal string→(ptr,len) in-wasm before the call.

**Summary of the audit.** The bulk of `src/server` (schema building, registration, BSATN ser/de, the module-def shape) is plain TS that is *compilable* once the backend exists, with `sys.*` host calls re-pointed from V8 externs to `spacetime_10.x` imports. The genuinely new code is small and concentrated: the wasm-ABI boundary (BytesSource/BytesSink drain loops, the two dunder export shims with exact signatures), a `spacetime_10.x` import-declaration file replacing `sys.d.ts`, and confirmation/trim of three JS-runtime dependencies (`object-inspect`, `url-polyfill`, `FinalizationRegistry`/`using`). Every one of these new pieces depends on the Perry `--target spacetimedb` backend, because re-pointing `sys.*` to a named wasm import and exporting a named dunder are exactly the two things Perry's `--target wasm` cannot express today (`crates/perry-codegen-wasm/src/emit/compile.rs:958,974` hardcode the import namespace to `"rt"`; FFI imports go under `"ffi"` per `module_emitter.rs:47`; exports are by-index `__wasm_func_<idx>` per `js_fallback.rs:311,653`).

## 3. The C#/C++ ABI-lowering pattern to mirror

C++ and C# are the existing precedent for "a non-Rust language lowered to the same native wasm ABI." Both reduce to the identical three-piece pattern, and the Perry path mirrors it exactly. Reading them tells the Perry backend what to emit and tells the TS-fork what glue to provide.

**Piece 1 — typed import declarations under `spacetime_10.x`.** C++ `abi.h:23-40` defines macros `STDB_IMPORT(name) = __attribute__((import_module("spacetime_10.0"), import_name(#name))) extern`, with sibling macros for `10.1`/`10.3`/`10.4`/`10.5`. Each host function is declared with its wasm signature using these attributes (e.g. `bytes_source_read`/`bytes_sink_write` at `abi.h:136-143`). `wasi_shims.cpp:10-13` shows the same attribute used standalone for `console_log`. This is the clang mechanism that pins the import module *and* field name — the direct analogue of what Perry's backend must add. The Perry equivalent of `abi.h` is the file that replaces `sys.d.ts`: a declaration set that the backend lowers to `import_module("spacetime_10.X")` + `import_name(...)` imports.

**Piece 2 — named dunder exports forwarding to a runtime impl.** C++ `module_exports.cpp:13-121` declares each export with `STDB_EXPORT(name) = __attribute__((export_name(#name)))` (`abi.h:202`) and a thin `extern "C"` function with the exact ABI signature — e.g. `__call_reducer__(uint32_t reducer_id, uint64_t sender_0..3, uint64_t conn_id_0..1, uint64_t timestamp_us, BytesSource args, BytesSink error)` at `module_exports.cpp:21-53` — that forwards to `Internal::Module::__call_reducer__`. C# does the identical thing in generated code: `bindings-csharp/Codegen/Module.cs:2799-2827` emits `[UnmanagedCallersOnly(EntryPoint = "__describe_module__")]` and `[UnmanagedCallersOnly(EntryPoint = "__call_reducer__")]` thin wrappers forwarding to `SpacetimeDB.Internal.Module`. The pattern is: a fixed, hand-shaped export-signature shim is *codegen*, the actual logic lives in a language-level runtime. For Perry, the export shim is the new-backend capability (export by name with chosen signature); the runtime logic is the re-pointed `runtime.ts`.

**Piece 3 — BSATN build + BytesSource/BytesSink drain loops in the runtime.** C++ `Module.cpp:387-397` (`__describe_module__`) serializes the module def to a `std::vector<uint8_t>` then `FFI::bytes_sink_write(sink, buffer.data(), &bytes_to_write)`. `Module.cpp:399-460` (`ConsumeBytes`) is the canonical args-read loop: query `bytes_source_remaining_length`, size the buffer, then `bytes_source_read` in a loop until exhausted (`ret == -1`). This is byte-for-byte the loop the TS Perry path must run before handing args to the existing `ProductType` deserializer — and the loop has no V8-side analogue in `runtime.ts` because the V8 host did it in Rust (`v2.rs:494-521`). C# Module.cs `__call_reducer__` (`Module.cs:521`) is the managed equivalent. **WASI satisfied in-module, never imported:** `wasi_shims.cpp` provides stub `__wasi_*` functions (e.g. `fd_write` redirected to `console_log`, `args_sizes_get`, `clock_*`) so the toolchain's libc dependencies resolve *inside* the module rather than as host imports — because the stock loader rejects any import outside `spacetime_10.x` (`02-stdb-abi.md` §ii). The Perry backend faces the same constraint: whatever runtime services Perry's lowered TS needs (its allocator/GC, string ops, Math/Date), it must satisfy in-wasm, not as `rt`/`ffi` host imports. This is doc 04's central wall restated as a build requirement: the Perry backend's job is to make Perry's runtime wasm-resident, the same way the C++ toolchain bundles libc and shims WASI.

**The reusable pattern, distilled:** (a) a typed import-declaration unit lowering to `import_module/import_name` imports under `spacetime_10.x`; (b) a small codegen shim exporting the dunders by exact name and signature, forwarding to (c) a language-level runtime that builds the BSATN module def, runs the BytesSource/BytesSink drain loops, decodes args, and dispatches by reducer id — with all toolchain runtime needs satisfied in-module. C++ and C# both instantiate this; the Perry path is the fourth instantiation, and (a)+(b) are exactly the new-backend capabilities, while (c) already exists in `runtime.ts` and only needs its `sys.*` calls re-pointed.

## 4. Embedding — how api-haus/perry is vendored into api-haus/SpacetimeDB

The fork relationship: `api-haus/SpacetimeDB` (fork of clockworklabs/SpacetimeDB) is where the CLI build branch and the re-pointed `crates/bindings-typescript/src/server` live; `api-haus/perry` (fork of PerryTS/perry) is where the `--target spacetimedb` backend is added. The TS-module fork invokes Perry as an external compiler at build time, exactly as `build_rust` shells out to `cargo` and `build_csharp` to MSBuild.

**Invocation: subprocess, mirroring the existing builders.** The new `build_javascript_perry` in `crates/cli/src/tasks/javascript.rs` runs the type-check (reuse the existing `tsc --noEmit` block, `javascript.rs:47-65`) and then, instead of the rolldown `Bundler` (`javascript.rs:73-221`), shells out to the Perry binary: `perry compile src/index.ts -o dist/module.wasm --target spacetimedb` (the new target). It returns `(project_path.join("dist").join("module.wasm"), )` and the `tasks/mod.rs` arm tags it `"Wasm"`. `tasks/mod.rs` already shells out to external tools via `duct::cmd` (it runs `wasm-opt` at `tasks/mod.rs:39`), so the dependency and idiom are present; the release Perry path can run the same `wasm-opt -O2` pass afterward.

**Vendoring choice — git submodule of the Perry fork (recommended), pinned binary as the fallback.**
- **Git submodule (recommended).** `api-haus/SpacetimeDB` carries `api-haus/perry` as a submodule at a pinned commit; CI builds the Perry binary from source and the CLI invokes it by path (or it is installed into the build image). This keeps the two forks co-versioned (the `--target spacetimedb` backend and the `spacetime_10.x` imports it emits must match the ABI major the TS runtime targets), makes the Perry source auditable from the SpacetimeDB fork, and is the natural home for the user's stated "the Perry fork may be embedded into the STDB-TS-module fork." It mirrors how SpacetimeDB already vendors toolchains.
- **Pinned prebuilt binary.** A `perry` binary pinned by version+sha512 and fetched into the build image. Lower build cost, but decouples the backend version from the ABI it emits — a silent-skew hazard. Acceptable for a frozen PoC, not for the product.
- **npm dependency.** Perry ships an npm wrapper (`@perryts/perry`), so the TS-module fork's `package.json` could depend on it. This co-locates with the TS tooling but pins to upstream PerryTS, not the `api-haus/perry` fork that has the new backend — so it does not work for the product until the fork's backend is published, and even then submodule-from-source is cleaner for a forked compiler under active change.

**One user-facing option, everything else identical.** The only surface the user sees is the `--engine perry` (or `--release-engine`) flag on `build`/`publish`. Dev `spacetime build` is byte-identical to today (rolldown→`bundle.js`, host_type `Js`, V8). Release `spacetime build --engine perry` produces `module.wasm`, host_type `Wasm`, stock Wasmtime. Same `src/index.ts`, same `tsc --noEmit` gate, same schema, same publish command.

## 5. Dev/release equivalence contract and PoC validation

**The contract.** For any module source `src/index.ts` that compiles under both paths:
1. **Same schema bytes.** `__describe_module__` emits byte-identical BSATN `RawModuleDef::V10` on both paths. Both paths run the same `registerReducer`/`schema()` pipeline producing the same `ctx.moduleDef`, then the same `RawModuleDef.serialize`. The describe bytes are a pure function of the schema object, so equality is checkable by dumping the module def from each published database (the byte layouts in `02-stdb-abi.md` §iv are the reference). This is the strongest single equivalence check and is the first PoC gate.
2. **Same reducer set and identity.** Same source names, same param `ProductType`, same `okReturnType = Product([])` / `errReturnType = String` (forced at `reducers.ts:83-84`), same reducer-id ordering (the id is the index into the `Reducers` section). The dev path's id assignment and the release path's must coincide because both derive from the same registration order.
3. **Same reducer semantics.** For identical args BSATN, the user reducer body runs the identical TS and produces the identical datastore mutations / checksum / error. The only legitimate difference is the engine (V8 JIT vs Perry AOT) and its performance; observable state must match.
4. **Same host_type routing aside.** Dev publishes `host_type=Js` (V8 loader); release publishes `host_type=Wasm` (Wasmtime loader). This is the one deliberate divergence and the whole point — both are first-class STDB module paths, unlike doc 04's proxy where the Perry arm ran under Node outside STDB entirely.

**What the PoC validates, and how it differs from doc 04.** Doc 04's benchmark was a *proxy*: the Perry arm ran the kernel under Node + Perry's `wasm_runtime.js`, on a different host process and clock from the V8-in-STDB arm, explicitly not a SpacetimeDB module. The equivalence contract here is only validatable once the Perry `--target spacetimedb` backend exists, and at that point the benchmark changes character entirely: **both arms run inside the same SpacetimeDB server, published from the same `src/index.ts`, differing only in host_type.** The validation:
- **Schema-equality gate (buildable the moment the backend emits a loadable module).** Publish the module both ways; dump each database's module def; assert byte-identical BSATN. If this fails, the paths are not equivalent and no latency comparison is meaningful.
- **Semantic-equality gate.** Call the same reducer with the same args on both databases; assert identical resulting state (the persisted checksum row, per the doc-04 `mix` kernel pattern, since reducers cannot return values — `reducers.ts:83`).
- **Latency comparison, now on a common clock.** Both arms expose the same server-measured `spacetime-execution-duration-micros` response header (doc 03/04: set from `finish_funcall` total_duration). Because both run in the same host, the header brackets the same stack (arg-deserialize + user fn), so the V8-JIT vs Perry-AOT ratio is a *fair* in-STDB comparison — the headline number doc 04 could not honestly produce. The §4-of-doc-04 caveats that still apply: Wasmtime fuel metering is ON for the Wasm arm (`02-stdb-abi.md` §iii) and the V8 budget is stubbed, so wall-clock is comparable but the energy stat is not; size the kernel under the fuel budget. Report median+IQR of the steady-state tail, cold separately (V8 has JIT warmup, Perry-AOT does not).

The equivalence contract is therefore the gate that turns doc 04's "codegen proxy" into the real experiment: same source, same schema, two published databases, one server, one clock.

## 6. Effort estimate by sub-task

The dominant cost is the Perry `--target spacetimedb` backend (the locked Option B), without which none of the equivalence contract is testable. The TS-fork surface work is small and mostly mechanical, but is downstream-blocked on the backend.

- **CLI flag + build branch (`build.rs` arg, `tasks/mod.rs` arm split, `build_javascript_perry` subprocess shim).** Self-contained, mirrors existing `--debug`/`build_csharp`. Days. Buildable and testable now against a hand-built Perry wasm via `--bin-path`, independent of the backend.
- **Re-pointing `src/server` (`sys.d.ts` → `spacetime_10.x` import-declaration unit; bind `sys.*` calls; BytesSource/BytesSink drain loops; trim/confirm `object-inspect`/`url-polyfill`/`FinalizationRegistry`).** Days-to-weeks once the backend exists; the drain loops and import-declaration unit are a direct port of the C++ `abi.h`/`ConsumeBytes` precedent. Cannot be end-to-end tested before the backend.
- **Embedding (Perry submodule into the SpacetimeDB fork, build-image/CI wiring, `wasm-opt` reuse).** Days.
- **Perry `--target spacetimedb` backend (the gating item): wasm-resident value representation + in-wasm GC/allocator; `import_module`/`import_name` import emission under `spacetime_10.x`; named dunder exports with the exact mixed-i32/i64 signatures; addressable-linear-memory primitive so BSATN bytes can be placed at sink/source offsets; in-module satisfaction of Perry's runtime services (no `rt`/`ffi` host imports).** This is the negative-space rewrite doc 04 described — replacing Perry's JS-host wasm backend, whose every avoided problem (wasm-resident values, addressable memory, chosen ABI) is exactly what must be added back. Multi-week, and the long pole. Effort signal for the whole effort is therefore **multi-week**, set by this item.
- **PoC validation harness (publish-both-ways, schema-equality dump+diff, semantic-equality call, in-STDB latency sweep).** Days, once a loadable Perry module exists.

## Side notes / observations / complaints

- **The host needs zero changes and the code proves it cleanly.** The entire "is this a Wasm or JS module" decision is a single `&'static str` returned from `tasks/mod.rs:30-34` and forwarded as a query param to `host_controller.rs:717`. A Perry-compiled module published as `host_type=Wasm` is indistinguishable from a Rust/C++/C# module to the server. The locked rejection of forking the host is well-founded: there is nothing to fork — the seam already exists.
- **The TS `ModuleHooks` interface is a JS-shaped facade, not the wasm ABI, and this is the single most important thing for an implementer to internalize.** `__describe_module__(): Uint8Array` and `__call_reducer__(..., argsBuf: DataView)` (`sys.d.ts:16-27`) look like the ABI but are not; the Rust V8 host does the BytesSource/BytesSink marshalling (`v2.rs:494-521`, `common.rs:126-139`). The Perry path has no host adapter, so the small-but-real new code is exactly the marshalling the V8 host hides. An implementer who ports `runtime.ts` literally will miss the drain loops entirely, because they are not in that file.
- **C++ is the cleanest precedent of the three** — `abi.h` + `module_exports.cpp` + `wasi_shims.cpp` + `Module.cpp` is a complete, readable instance of "compile a non-Rust language to the STDB native wasm ABI," including the in-module-WASI trick that is the literal answer to "what do you do about the toolchain's host-import needs." It should be the reference doc for the Perry backend author. C# is the same pattern but buried in Roslyn codegen.
- **`reducers.ts:82-84` hardcodes no-return-value reducers** (`okReturnType = Product([])`). This is load-bearing for the equivalence contract: both paths inherit the same no-return shape, so neither the dev nor release arm can return a value, and the benchmark must persist+read a checksum (as doc 04 already does). When upstream eventually adds reducer return values, both paths must track it together.
- **The benchmark only becomes honest after the backend exists.** Everything in §5 about a common clock and a fair ratio is contingent on a loadable Perry module. Until then, doc 04's proxy is the only buildable measurement and its "not a common clock / not in STDB" caveats stand. The work that closes that gap is the multi-week Perry backend, not the TS-fork surface — the surface designed here is small and would land in days if the backend were free.
- **Perry's wasm backend at HEAD still hardcodes the JS-host shape** (`perry-codegen-wasm/src/emit/compile.rs:958,974` import namespace `"rt"`; `module_emitter.rs:47` FFI under `"ffi"`; `js_fallback.rs` by-index `__wasm_func_<idx>` exports; `wasm_runtime.js` NaN-boxing). I re-confirmed this against the vendored Perry source rather than trusting doc 04's reproduction; it holds. The `perry-wasm-host` crate uses `instantiate_and_start` (command module), further confirming the artifact is not the reactor shape STDB needs.
